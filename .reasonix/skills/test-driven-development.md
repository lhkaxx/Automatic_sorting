# Test-Driven Development (TDD)

## 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

在测试之前写了实现代码？删除它。重新开始。没有例外。

## RED-GREEN-REFACTOR 循环

### RED — 写一个失败的测试

写一个最小测试，展示应该发生什么。

**要求：**
- 只测一个行为
- 测试名清晰描述行为（如 `test('rejects empty email')`）
- 用真实代码（非必要时不用 mock）
- 断言先写：先想清楚期望结果

### Verify RED — 必须看它失败

```bash
# 运行测试，确认：
# 1. 测试失败了（不是报错）
# 2. 失败信息符合预期
# 3. 是因为功能缺失而失败（不是拼写错误）
```

测试直接通过了？你在测已有行为 → 修改测试。
测试报错了？修复错误，重新运行直到正确失败。

### GREEN — 最小代码通过

只写让测试通过的最少代码。不要：
- 加未来才需要的参数
- 重构无关代码
- "顺便改进"任何东西

### Verify GREEN — 必须看它通过

```bash
# 确认：
# 1. 新测试通过
# 2. 所有旧测试依然通过
# 3. 输出干净（无 error、warning）
```

### REFACTOR — 清理

只有在全绿之后才能：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿色。不加新行为。

## 好测试 vs 坏测试

| 维度 | 好 | 坏 |
|------|-----|-----|
| 最小化 | 只测一件事。名字里有"和"就拆分 | `test('validates email and domain')` |
| 清晰 | 名字描述行为 | `test('test1')` |
| 展示意图 | 演示期望的 API | 隐藏代码应该做什么 |

## 常见借口粉碎

| 借口 | 现实 |
|------|------|
| "太简单不需要测" | 简单代码也会坏。30 秒写一个测试。 |
| "我先写代码再补测试" | 后补的测试直接通过，证明不了任何事。 |
| "已经手动测过了" | 手动 = 临时。没有记录，不能重跑。 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误。保留不可信的代码才是技术债。 |
| "这是务实，不是教条" | TDD 本身就是务实的：提交前发现 bug，防止回归，文档化行为，支持重构。 |

## 红灯 — 立即停手

以下任何一种情况出现 → 删除代码，用 TDD 重来：
- 先写代码再写测试
- 测试一次性直接通过
- 说不清测试为什么失败
- "这次就算了" 的念头
- "先保留代码作为参考"

## Bug 修复流程

发现 bug → 先写一个复现它的失败测试 → 走完整 RED-GREEN-REFACTOR → 测试证明修复有效且不会回归。

绝不无测试修 bug。

## 完成前检查清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试都看过它先失败
- [ ] 每个测试因正确原因失败（功能缺失，不是拼写错误）
- [ ] 每个测试只写了最小代码通过
- [ ] 全部测试通过
- [ ] 输出干净
- [ ] 边界情况和错误都已覆盖
