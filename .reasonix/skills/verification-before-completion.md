# Verification Before Completion — 完成前自检

## 何时使用

在声明"做完了"之前，必须逐条自检。适用于：
- 完成一个模块/文件
- 完成一个 feature
- 修复一个 bug
- 完成一个计划步骤

## 核心原则

**你说"做完了"之前，先用证据证明它真的做完了。**

## 自检清单

### 1. 功能正确性

- [ ] 对照规范/需求逐条核对，每一条都能在代码中找到对应实现
- [ ] 如果是 bug 修复：确认有测试覆盖该 bug，且测试先失败后通过
- [ ] 所有正常路径走通
- [ ] 所有边界情况和错误路径有处理（空输入、超限、异常数据）

### 2. 测试

- [ ] 每个新函数/方法有对应测试
- [ ] 每个测试都看过它先失败再通过（TDD）
- [ ] 全部测试通过：`pytest tests/ -v`（或用项目实际的测试命令）
- [ ] 没有跳过的测试（skip/xfail 必须有注释说明原因）
- [ ] 输出干净，无 error、warning

### 3. 约束合规

- [ ] 对照项目禁止事项逐条检查
- [ ] 没有引入被禁止的依赖/框架/模式
- [ ] 数据模型关系正确（如一对一/一对多）
- [ ] 硬约束未被违反

### 4. 代码质量

- [ ] 没有硬编码的测试数据
- [ ] 没有注释掉的代码（要么删除，要么恢复并加注释说明）
- [ ] 命名清晰一致
- [ ] 没有 `print` 调试残留
- [ ] import 都是实际用到的

### 5. 集成

- [ ] 如果是新文件，已在父模块中 import/注册
- [ ] 如果是新路由，已确认不和现有路由冲突
- [ ] 如果修改了数据模型，已确认下游代码不受破坏
- [ ] 如果新增了依赖，已更新 requirements.txt / package.json

### 6. 文档

- [ ] 如果新增了公开 API，已更新 API 文档
- [ ] 如果修改了配置项，已更新配置说明
- [ ] 如果有新参数，已标注默认值和含义

## 验证方法

对每一项，不要凭记忆回答——用工具验证：

```
"测试全过" → 运行 pytest 并看输出 ✓
"路由正确" → curl 验证或检查 app.py 中的注册 ✓
"没忘 import" → 运行主入口看是否报 ImportError ✓
"数据模型正确" → 跑 test_models.py ✓
```

## 红牌 — 以下情况绝不能声明完成

- 有测试失败
- 有硬约束违规
- 有已知 bug 未修复
- 有 `# TODO` 或 `# FIXME` 未处理（除非有明确理由并记录）
- 有 `print` / `console.log` 调试残留
- 有被注释掉的代码块

## 输出格式

完成自检后输出：

```markdown
## 自检结果

### ✅ 通过
- 功能正确性：对照规范第 X 节，全部实现
- 测试：12 passed, 0 failed
- 约束合规：无违规
- ...

### ⚠️ 注意事项
- xxx（如果有需要说明的点）

### ❌ 未通过（阻塞交付）
- xxx（如果有）
```
