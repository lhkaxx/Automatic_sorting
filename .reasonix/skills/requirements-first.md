# Requirements-First Development — 需求先行开发流程

## 铁律

```
NO CODE WITHOUT APPROVED REQUIREMENTS DOCUMENT
```

需求文档未被用户确认之前，禁止写任何代码。没有例外。

---

## 三阶段流程

```
用户提出主题
    ↓
┌─────────────────────────────┐
│ 阶段一：主题确认（问询）      │
│ • 逐维度向用户确认细节        │
│ • 覆盖：功能、数据、界面、约束 │
│ • 用户确认完毕 → 进入阶段二   │
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│ 阶段二：需求文档编写          │
│ • 输出完整的 MD 需求文档      │
│ • 包含：项目概览、数据模型、    │
│   功能列表、界面描述、约束、    │
│   技术栈、测试要求             │
│ • 用户审阅确认 → 进入阶段三   │
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│ 阶段三：代码实现              │
│ • 按需求文档逐模块实现         │
│ • 每模块：写测试 → 写代码      │
│ • 完成后自检                   │
└─────────────────────────────┘
```

---

## 阶段一：主题确认（问询清单）

当用户提出一个开发主题后，**必须逐维度确认以下内容**。不要一次性全问——每次 2-3 个问题，分轮推进。

### 1. 项目定位
- 这个系统/工具给谁用？（用户画像）
- 核心解决的问题是什么？（一句话）
- 有没有类似的参考产品？（竞品/参考）

### 2. 功能范围
- 核心功能有哪些？（必须有的）
- 扩展功能有哪些？（锦上添花的）
- 明确不做什么？（画边界，防止范围膨胀）

### 3. 数据模型
- 系统涉及哪些实体？（人、物、关系）
- 每个实体有哪些字段？哪些是必须的？
- 实体之间的关系是什么？（一对一/一对多/多对多）
- 数据从哪里来？（用户输入 / 文件上传 / API / 数据库）

### 4. 输入输出
- 用户如何输入数据？（表单 / CSV / Excel / API / 其他）
- 输入格式具体是什么？（列名、编码、必填项）
- 系统输出什么？（页面展示 / 文件导出 / API 返回）
- 输出格式是什么？（JSON / CSV / PDF / 图表 / 其他）

### 5. 界面与交互
- 有几个页面？页面之间的流转关系？
- 每个页面的核心功能是什么？
- 有无特殊的交互需求？（拖拽 / 实时更新 / 动画 / 搜索 / 过滤）
- 移动端需要吗？响应式要求？

### 6. 约束与规则
- 有哪些硬性规则？（必须满足，违规则无效）
- 有哪些软性规则？（尽量满足）
- 有哪些禁止事项？（不能做的）

### 7. 技术栈
- 后端用什么？前端用什么？
- 有无必须使用的库/框架？
- 有无明确禁止使用的技术？

### 8. 测试与交付
- 测试要求？（单元测试 / 集成测试 / 覆盖率）
- 交付物除了代码还有什么？（文档 / 部署说明 / 示例数据）
- 有无时间/性能约束？

### 问询规则
- 每次只问 2-3 个问题，不要轰炸
- 用户已回答的不再重复问
- 如果用户说"随便"或"你定"，给出推荐方案并说明理由
- 每轮结束时总结已确认内容，标注待确认项

## 阶段二：需求文档编写

全部确认完毕后，生成 MD 格式需求文档。结构如下：

```markdown
# [项目名称] — 全栈开发规范

## 一、项目概览
- 1.1 项目定位
- 1.2 技术栈
- 1.3 项目结构

## 二、数据模型
- 每个实体的 dataclass 定义（字段、类型、说明）
- 实体关系图

## 三、核心功能
- 3.1 功能列表（优先级标注）
- 3.2 功能详细描述

## 四、输入输出规范
- 4.1 输入格式（CSV/表单/API 格式）
- 4.2 输出格式

## 五、界面设计
- 5.1 页面流程图
- 5.2 每个页面的布局和组件描述
- 5.3 UI 风格（配色/字体/组件形式）

## 六、约束与规则
- 6.1 硬约束
- 6.2 软约束
- 6.3 禁止事项

## 七、后端 API（如适用）
- 路由表

## 八、测试要求
## 九、错误处理与边界情况
```

生成后 **必须等待用户确认**。用户可以：
- "可以" / "开始" → 进入阶段三
- 提出修改意见 → 修改需求文档后重新确认

---

## 阶段三：代码实现

需求文档确认后，调用 `writing-plans` 技能拆分任务，然后逐任务实现。每个任务遵循 `test-driven-development` 技能。

---

## 特殊情况处理

### 用户提供了参考文档（如 docx/pdf）
- 先读取文档内容
- 提取关键需求信息
- 对照问询清单，对文档中未覆盖的维度进行补充确认
- 将文档内容 + 补充确认合并为完整需求文档

### 用户中途修改需求
- 如果还在阶段一/二：直接更新
- 如果已经进入阶段三：评估影响范围，标注"已实现 vs 需返工"，请用户确认后再改

### 范围膨胀
- 用户在阶段一确认"不做 X"，阶段二又想加 X → 重新确认优先级和影响
- 需求文档确认后用户想加功能 → 在需求文档中追加，标注为"新增"，单独评估
